01 What Problem Does This Solve?

Managing consistent AI workflows across a growing portfolio of projects.

When multiple projects use Claude Code, each needs hooks, guards, skills, agents, and configuration. Without a central system, these diverge quickly — a guard fix in one project never reaches the others, skill improvements stay local, and onboarding new projects means copy-pasting from the most recent one.

claude-workflow solves this by acting as a canonical master repo. Projects initialize from it, pin to a version, and receive drift detection when the master evolves. Each project can exclude files it owns locally while staying in sync on everything else.

1
Centralized

Single source of truth for all workflow configuration. Fix a guard once, roll it out everywhere.

2
Versioned

Semantic versioning with version pinning. Projects upgrade deliberately, never accidentally.

3
Stack-Aware

Technology-specific overlays. TypeScript projects get different guards than Python or Flutter.

4
Customizable

Projects exclude files they own and merge additional settings without breaking sync.

At initialization, base files are copied first. Stack overlays are then applied on top — hooks and guards are appended, not replaced. Finally, project overrides let each repo exclude files it owns locally and merge additional settings.

03 The Sync Lifecycle

Three tools, three phases: initialize, detect, update.

Phase 1
init.sh
Copy base & stack files, resolve placeholders, generate lock
Phase 2
diff.sh
Three-way checksum comparison: local vs lock vs master
Phase 3
sync.sh
Auto-update BEHIND files, restore MISSING, warn on DIVERGED

Phase 1: Initialization

Terminal
./tools/init.sh --project /path/to/my-project --stacks typescript-nestjs

This command performs five steps in sequence:

1
Copy Base
Hooks, skills, agents, blueprints
2
Apply Stacks
Stack hooks, guards, failure patterns
3
Resolve
{{PLACEHOLDER}} values from commands.yaml
4
Compose
Merge settings: base + guards + overlays
5
Lock
Generate workflow.lock with checksums

Phase 2: Drift Detection

Terminal
./tools/diff.sh --project /path/to/my-project

Compares three checksums per file to produce a status:

Local vs LockMaster Changed?StatusAction
match no CURRENT No action needed
match yes BEHIND Auto-update available
differ yes DIVERGED Manual merge needed
differ no LOCAL-EDIT Project modified (allowed)
file deleted locally MISSING Restore from master

Phase 3: Sync Updates

Terminal
./tools/sync.sh --project /path/to/my-project --auto

Automatically updates BEHIND files and restores MISSING files. Reports DIVERGED files for manual attention. LOCAL-EDIT files are left untouched. After sync, the lock is regenerated with current checksums.

Each stack provides a commands.yaml file that defines placeholder values. These are resolved into all .md files during initialization, so project files contain concrete commands rather than generic placeholders.

05 Placeholder Resolution

Master files use {{PLACEHOLDER}} syntax. Stacks provide the values.

Master template files (skills, blueprints, WORKFLOW.md) reference commands generically. During init.sh, all .md files are walked and placeholders are replaced with concrete values from the stack's commands.yaml.

Before (master source)
Run: {{FORMAT_COMMAND}}
Tests: {{TEST_COMMAND}}
Version: {{VERSION}}
After (project file)
Run: pnpm nx affected -t lint --fix
Tests: pnpm nx affected -t test
Version: 1.0.0
How it works under the hood: The init script collects all key-value pairs from each stack's commands.yaml (later stacks override earlier ones). List values like classify_categories are joined with , . Two special values — {{VERSION}} and {{STACKS}} — are always available.

06 The Lock File

workflow.lock is the contract between a project and the master version it synced from.

workflow.lock
{
  "version": "1.0.0",
  "stacks": ["typescript-nestjs"],
  "lastSync": "2026-03-03T14:24:12Z",

  // Checksums of LOCAL files (after placeholder resolution + formatting)
  "managed": {
    "WORKFLOW.md":           "sha256:6e7193ff...",
    "hooks/session-start.sh": "sha256:abc123...",
    "settings.json":          "sha256:def456..."
  },

  // Checksums of RAW MASTER source files (pre-resolution, pre-formatting)
  "masterChecksums": {
    "WORKFLOW.md":           "sha256:a1b2c3...",
    "hooks/session-start.sh": "sha256:abc123...",
    "settings.json":          "sha256:def456..."
  }
}

Two sets of checksums serve different purposes:

FieldContainsPurpose
managed Hash of local file (post-resolution, post-formatting) Detect local edits: has the project modified this file since sync?
masterChecksums Hash of raw master source (pre-resolution) Detect master changes: has the master evolved since sync?
Why two sets? A project's local file may differ from the raw master source for legitimate reasons: placeholder resolution turns {{FORMAT_COMMAND}} into a real command, and formatters like prettier may reformat whitespace. Storing both hashes lets drift detection compare apples to apples — local-to-local and master-to-master.

07 Three-Way Drift Detection

The core algorithm that determines the status of every managed file.

For each file tracked in the lock, drift_check.py computes two boolean comparisons and maps them to one of five statuses:

drift_check.py — core logic
# 1. Has the local file changed since last sync?
local_hash = sha256(local_path)
local_matches_lock = local_hash == lock["managed"][rel_path]

# 2. Has the master source changed since last sync?
if rel_path in lock["masterChecksums"]:
    # Compare raw master hashes (formatter-resilient)
    current_master = sha256(master_path)
    master_changed = lock["masterChecksums"][rel_path] != current_master
else:
    # Fallback for pre-upgrade locks: compare with resolution
    master_hash = sha256_resolved(master_path, commands)
    master_changed = lock_hash != master_hash

# 3. Map to status
if     local_matches_lock and not master_changed:  # CURRENT
elif   local_matches_lock and     master_changed:  # BEHIND
elif not local_matches_lock and   master_changed:  # DIVERGED
else:                                              # LOCAL-EDIT

Excluded Files

Projects can mark files as excluded in workflow.overrides.yaml. Excluded files are skipped entirely during drift detection — the project owns them.

workflow.overrides.yaml
exclude:
  - skills/debug/
  - skills/generate-endpoint/
  - agents/architecture-guardian.md

08 The Formatter Problem & Fix

Why masterChecksums was added and the problem it solves.

The Problem

Some projects run formatters (like prettier) as pre-commit hooks. When a project initializes from master, the files are copied and placeholders resolved. But on commit, prettier reformats the .md files — changing whitespace, line lengths, and list formatting.

Before the fix

The lock stored one set of hashes (local files post-prettier). Drift detection re-computed master hashes by resolving placeholders but without running prettier. These never matched → BEHIND on every file, every time.

After the fix

The lock stores two sets of hashes. Drift detection compares raw master → raw master (same space, no formatter involved). Local comparison uses post-formatter hashes. Each comparison stays in its own domain.

The Timeline


  init.sh runs
    master/base/WORKFLOW.md  ──resolve──>  project/.claude/WORKFLOW.md
                                                    │
  git commit triggers prettier                     │
                                                    ▼
                                          WORKFLOW.md (reformatted)

  workflow.lock records:
    managed[WORKFLOW.md]         = sha256(reformatted file)   ← post-prettier
    masterChecksums[WORKFLOW.md] = sha256(raw master source)  ← pre-resolution

  diff.sh runs later:
    local hash                   = sha256(reformatted file)   ← matches managed
    current master hash          = sha256(raw master source)  ← matches masterChecksums
                                                                      ▼
                                                               CURRENT
Backward compatibility: Locks generated before this fix lack masterChecksums. The drift checker falls back to the old behavior (resolving placeholders and comparing against lock hashes) for these files. Projects can regenerate their lock via sync.sh to upgrade.

09 CI/CD Integration

Automated drift checks across 9 projects and release management.

CI
drift-check.yml

Runs weekly (Monday 8am UTC), on release, and on manual trigger. Matrix strategy checks all 9 projects in projects.json in parallel. If drift is found, creates an issue in the project repo with label workflow-drift.

CD
release.yml

Triggered when version.json changes on main. Creates a git tag and GitHub release with auto-generated notes. Projects upgrade by pinning the new version.

Project Registry

The workflow uses a matrix strategy driven by projects.json to check all projects in parallel. A WORKFLOW_PAT secret (fine-grained token with contents:read) enables cross-repo checkout.

CI Workflow Architecture

drift-check.yml
on:
  schedule:   [cron: '0 8 * * 1']  # Weekly Monday 8am UTC
  workflow_dispatch:                  # Manual trigger
  release:   [types: published]        # On new version

jobs:
  load-projects:                       # Read projects.json
    outputs: projects matrix

  check-drift:                         # Parallel matrix (9 jobs)
    strategy:
      matrix: ${{ fromJson(projects) }}
    steps:
      - checkout master repo
      - checkout project repo (via WORKFLOW_PAT)
      - diff.sh --project project --json
      - if BEHIND|DIVERGED: create issue

Per-Project Drift Workflow

Each project also has its own .github/workflows/workflow-drift.yml installed during onboarding. This enables projects to run local drift checks independently.

Version Upgrade Flow

# 1. Master publishes a new version
#    (bump version.json → release.yml creates tag v1.1.0)

# 2. CI drift check detects the gap, opens issue in each project
#    title: "Workflow drift detected against claude-workflow master"

# 3. Project upgrades explicitly
./tools/sync.sh --project /path/to/erp --version 1.1.0

# 4. Verify zero drift
./tools/diff.sh --project /path/to/erp
# Workflow Drift Report (pinned: v1.1.0)
#   CURRENT: 55 files
No surprise upgrades. Projects pin to a specific version in their lock file. New master releases do not auto-deploy. The CI drift check detects when a project falls behind and notifies via GitHub issues, but the upgrade is always an explicit action.
