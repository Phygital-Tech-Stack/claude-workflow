# Teams: First-Class Orchestration in claude-workflow

**Status**: Complete
**Date**: 2026-03-10
**Author**: Brainstorm session

## Problem Statement

The ERP project has a working `.claude/teams/` implementation that enables parallel agent orchestration for validation and code generation. This pattern is project-specific and not available to other managed projects. We need to promote teams to a first-class concept in the master workflow so all 11 projects can benefit from coordinated parallel agent workflows.

## Design Decisions (Confirmed)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Primary goal | Both orchestration + prompt sharing + new workflows | Teams as a full first-class concept |
| Relationship to agents | Teams **reference** agents | Industry best practice; agents stay independent and reusable |
| Relationship to official Agent Teams | **Complement** the official runtime | We provide project-level prompt templates and roster configs; official feature handles execution |
| Base vs stack scope | Validation team in base; generation teams are stack-specific | Validation is universal; generation varies by tech stack |
| Sync strategy | Merge base + stack teams (same as hooks/guards) | Consistent with existing sync patterns |
| Config depth | Flat structure + thin manifest (`team.yaml`) | Mirror working ERP pattern; add machine-discoverable manifest |

### team.yaml Schema (Minimal)

```yaml
name: validation
description: Multi-lens code review team for parallel validation
members:
  - code-reviewer        # references base/agents/code-reviewer.md
  - security-reviewer    # references base/agents/security-reviewer.md
  - arch-checker         # team-only specialist (prompt in prompts/)
```

**Rules:**
- Members that match a `base/agents/*.md` name are references to standalone agents
- Members without a matching agent definition must have a prompt in `prompts/<name>.md`
- No orchestration logic in team.yaml — skills own execution strategy
- Model hints live in each prompt file's frontmatter (or inline), not in team.yaml

### Relationship to Existing Agents

```
base/agents/                      base/teams/validation/
  code-reviewer.md  <----ref----    team.yaml (members: [code-reviewer, ...])
  security-reviewer.md <-ref----    prompts/
  planner.md                          arch-checker.md  (team-only specialist)
  backend-handler.md
  frontend-handler.md
  test-writer.md
  db-expert.md
```

- Agents remain standalone in `base/agents/`
- Teams reference agents by name
- Teams can define **team-only specialists** via `prompts/<name>.md`
- An agent can appear in multiple teams
- Team prompts can add team-specific context on top of the base agent definition

### Prompt File Format

Team prompts follow the same pattern as the ERP implementation — self-contained markdown with:

```markdown
# <Role> Teammate

You are a <role description>.

## Your Task
<what this teammate does>

## Context
- Read `CLAUDE.md` for project rules
- Read `.claude/agents/<name>.md` for the base checklist (if referencing an agent)
- Reference <golden path> as the reference implementation

## Checks / Rules
<domain-specific checklist>

## Output Format
<exact output template>

## Important
<guardrails>
```

### Sync Engine Changes

**init.sh** — After copying `base/` and `stacks/{stack}/`, also:
1. Copy `base/teams/` to `.claude/teams/`
2. Copy `stacks/{stack}/teams/` to `.claude/teams/` (merge, stack wins on conflict)
3. Add team files to `workflow.lock` checksums

**sync.sh** — Same three-way drift detection for team files as for hooks/guards/skills.

**workflow.overrides.yaml** — Teams can be excluded:
```yaml
exclude:
  - teams/generation/prompts/custom-builder.md
```

### Cross-Cutting Refactor Pattern (from ERP)

Teams also enable a powerful refactor pattern documented in the validation team's README:

1. Lead defines the change pattern with before/after examples
2. List affected modules via Glob/Grep
3. Spawn one teammate per module (Task tool, `subagent_type="general-purpose"`)
4. Each teammate applies the pattern to their assigned files
5. Lead verifies consistency
6. Run `/validate-change --team` on combined changeset

This pattern ships in the base `validation/README.md` as documentation.

## Skill Integration

### `/validate-change --team` (update existing skill)

The existing validate-change skill's Layer 4 (Agentic) currently spawns a single code-reviewer. With teams:

1. Read `.claude/teams/validation/team.yaml` to discover members
2. For each member, read the prompt from `prompts/<name>.md` (or fall back to `agents/<name>.md`)
3. Spawn parallel Task calls, one per teammate
4. Aggregate results: overall verdict = worst individual verdict

### `/generate-module --team` (new stack-specific skill)

Each stack that ships a generation team also ships a corresponding `/generate-module` skill. The skill:

1. Reads the design doc / user input
2. Produces a Module Blueprint (structured data all teammates receive)
3. Reads `.claude/teams/generation/team.yaml` for the roster
4. Spawns parallel Tasks with blueprint + teammate prompt
5. Assembles outputs, registers the module, runs tests

### Future: Official Agent Teams Integration

When the official Agent Teams feature stabilizes:
- team.yaml can be extended with an `agent-teams` section for runtime config
- Skills can choose between Task-based subagents (current) and official Agent Teams (future)
- The prompt files remain the same regardless of execution backend

## Migration Path

1. **v1.4.0**: Ship base validation team + typescript-nestjs generation team
2. Projects run `/sync-workflow --update` to receive teams
3. ERP migrates from custom `.claude/teams/` to synced version (add overrides for ERP-specific customizations)
4. Other stacks add generation teams incrementally

## Out of Scope

- Dynamic team composition at runtime
- Team-level permissions or quotas
- Inter-teammate messaging (handled by official Agent Teams if needed)
- Automatic team selection (skills explicitly choose which team to invoke)
